Estructura del repositorio tld-validador-proxy

Un único repositorio alinea aplicación e infraestructura SAM con el estándar de servicios como tld-notificacion y tld-api-p2m: carpeta lambdas/layer para dependencias Node, CodeUri bajo lambdas/proxy con lib/ colateral, y sin API Gateway — el borde productivo es invocación directa (lambda:InvokeFunction), con opción de Capa 2 vía SQS + worker. Así el proyecto se reconoce como servicio interno de alta resiliencia y baja latencia de red de entrada.

Árbol de directorios

El CodeUri de las funciones Lambda apunta a lambdas/proxy; la capa SAM empaqueta lambdas/layer (convención nodejs/). 

tld-validador-proxy/
├── events/                 # Payloads JSON para pruebas de sam local invoke
├── lambdas/
│   ├── proxy/              # Directorio principal (CodeUri)
│   │   ├── orchestrator.js # Handler: invocación síncrona (Capa 1)
│   │   ├── worker.js       # Handler: trigger SQS (Capa 2)
│   │   └── lib/            # Lógica compartida (httpsClient, dynamoDB, resilience, …)
│   └── layer/              # Capa de dependencias (node_modules)
│       └── nodejs/
│           └── package.json
├── template.yaml           # 2 funciones, cola SQS, DLQ (sin API Gateway)
└── samconfig.toml          # Perfiles [dev], [sandbox], [qa], [prod] + parameter_overrides

Los nombres de handler en SAM deben coincidir con los puntos de entrada exportados (p. ej. Handler: orchestrator.lambdaHandler y Handler: worker.lambdaHandler).

Especificaciones de diseño

Sin API Gateway

Los satélites autorizados invocan la función de Capa 1 con el SDK de AWS (lambda:InvokeFunction). Se reduce latencia de borde, se elimina la necesidad de VPC endpoint dedicado al API Gateway para exponer el proxy, y la superficie IAM queda explícita en el rol del llamador. Detalle operativo: Infraestructura SAM — invocación SDK.

Importación de módulos

Los handlers deben resolver el código compartido con rutas relativas al archivo del handler, por ejemplo require('./lib/httpsClient') o require("./lib/resilience"), igual que en tld-notificacion y tld-api-p2m. SAM incluye todo el árbol bajo CodeUri; las dependencias npm pesadas viven en lambdas/layer/nodejs tras npm install en esa carpeta.

Gestión de entornos (samconfig.toml)

Se recomienda replicar el esquema de perfiles usado en tld-api-p2m: secciones [dev], [sandbox], [qa], [prod] (y, si aplica, variantes regionales tipo [dev-oregon]) con stack_name, region, s3_bucket / s3_prefix y una cadena parameter_overrides que inyecte parámetros del template.yaml: VPC vía Subnet1..3, SecurityGroup, montaje EFS (AccessPointId, EFSResourceId), rutas CA/CRL, DeployEnvironment, y demás claves definidas en la plantilla. Guía de red: Red (samconfig).

Lógica de resiliencia — lambdas/proxy/lib/resilience.js

Este módulo concentra el time budget de la invocación (p. ej. en combinación con context.getRemainingTimeInMillis() y los límites del SDK del llamador): decide si un fallo reintentable se resuelve en la misma ejecución síncrona (Capa 1) o si conviene delegar el trabajo a la cola SQS para que worker.js (Capa 2) continúe con backoff y política DynamoDB. Así se mantiene la resiliencia controlada documentada en la wiki sin obligar a un único camino síncrono. Contexto funcional: Arquitectura SQS y Capa 2 y Configuración en DynamoDB.

Relación con template.yaml

La plantilla define típicamente: función de orquestación (sin evento Api), función worker con evento SQS, cola de resiliencia y DLQ, políticas IAM mínimas, FileSystemConfigs para /mnt/tld-llaves, y Outputs con ARN/nombre de la Lambda invocable. Borrador completo: Infraestructura SAM — template propuesto.
