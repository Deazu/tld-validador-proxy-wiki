Misión del tld-validador-proxy

Soberanía tecnológica y seguridad proactiva para la validación de canales

Portal técnico: un solo punto de salida controlado hacia validadores — cifrado híbrido correcto, Conectividad Segura (TLS / mTLS) con CA y CRL en EFS, identificación del proxy ante el validador mediante certificado de cliente en TLS / mTLS cuando el canal lo exige, identidad de negocio en DynamoDB y trazas sin filtrar secretos.

Lectura recomendada para stakeholders

¿Por qué este proyecto? — Valor y justificación

Además del contexto legacy (tld-api-cuenta-nombre, tld-util, bifurcación axios / CA Telered) y del cifrado de aplicación centralizado, la página concentra tres pilares que explican la Lambda:

  • Corrección del flujo de validación CRL (Seguridad proactiva: revocación antes del envío del body).
  • Identidad soberana del Proxy (TLS / mTLS) (certificado de cliente mTLS hacia el validador cuando el canal lo exige, perfil por canal en DynamoDB).
  • Dashboard de Salud de Canales (Observabilidad forense por fase de Conectividad Segura TLS y mTLS, CRL, HTTP y PROXY_ERR_*).

Profundiza en hallazgos, diagramas comparativos y telemetría en el documento enlazado.

Ir a valor y justificación

Nuestra Misión: Garantizar la integridad y confidencialidad en el transporte de datos financieros, eliminando la deuda técnica de validaciones reactivas y estableciendo una identidad digital soberana mediante Conectividad Segura TLS / mTLS (autenticación mutua donde el canal lo exige) y observabilidad forense alineada a Validación de Confianza Cero (Zero Trust) en el perímetro hacia validadores.

Gestión de disponibilidad: El Proxy incorpora un Sistema de Resiliencia Opt-in (bajo demanda): gobernado por DynamoDB — nada es automático si no está definido en tabla — con políticas explícitas, cola SQS y worker cuando el despliegue lo habilite. La orquestación y la configuración operativa se documentan en Estrategia y Capa 1, SQS y Capa 2 y Manual de configuración. Contexto de negocio y auditoría legacy: Valor y justificación — §4.

El proxy es el único punto de salida controlado hacia los canales validadores remotos: centraliza cifrado híbrido (RSA/KMS + AES), Conectividad Segura (TLS / mTLS) con CA en EFS, comprobación de CRL y la resolución de identidad desde DynamoDB, de modo que los microservicios satélite no repitan ni omitan parámetros críticos. No solo valida al validador con la cadena corporativa: cuando el canal lo define, el proxy presenta su propio certificado de cliente para que el socio remoto autentique nuestra salida igual que nosotros autenticamos la suya.

Secuencia maestra: orquestación + telemetría lateral

Cada fase crítica deja huella en Telemetría (DynamoDB): no bloquea el camino feliz si falla el PutItem (política fail-safe), pero en condiciones normales reconstruyes el incidente como si leyeras la cinta negra junto al portero. El borde de entrada al proxy es Consumer Service → AWS SDK (Invoke) → función Lambda privada (ARN del Proxy o nombre de la función) — ver Estructura del repositorio, Infraestructura SAM y Flujo General.

      Note over Htt,Rem: TLS y mTLS cert cliente si canal
      Htt->>Rem: POST body cifrado
      Rem-->>Htt: respuesta HTTP
      Htt->>Tel: emit HTTP_Response
      Htt-->>Op: bytes y status
    else Path_FAIL_CRL
      Htt->>Htt: ABORTA_sin_body_validador
      Htt->>Tel: emit error PROXY_ERR
      Htt-->>Op: error controlado
    end
    Op-->>Log: abrir paquete o error
    Log-->>Sdk: payload retorno Lambda
    Sdk-->>Cons: resultado Invoke

La Lambda actúa como director de orquesta: IdentityManager sustituye el legado de canal.js con lecturas GetItem/GSI y DTO congelado (nadie reescribe el pasaporte a mitad del control); httpsClient concentra el portero de Conectividad Segura (TLS / mTLS), la CRL y el doble pasaporte TLS / mTLS: el validador prueba su identidad hacia nosotros y, cuando el perfil del canal lo exige, nosotros presentamos certificado de cliente hacia él. Entre ambos, OperacionesPaquete llena las cubetas de cifrado de aplicación antes de que un solo byte sensible cruce el cable hacia el validador. El porqué estratégico del proyecto (auditoría, riesgos legacy, centralización de cifrado y telemetría) está concentrado en Valor y justificación.

Alcance

  • Orquestación: dos lecturas de canal (emisor / validador), cifrado hacia el validador remoto (HTTPS), con política unificada de Conectividad Segura (TLS / mTLS).
  • Telemetría técnica en DynamoDB (tabla dedicada) para handshake, CRL, capa TLS / mTLS, HTTP y tiempos.
  • Compatibilidad de despliegue con el estándar SAM de los repos existentes (VPC, EFS, KMS MRK).

Stack tecnológico

  • Runtime: Node.js 20.x (LTS)
  • AWS SDK v3 — clientes modulares (@aws-sdk/client-dynamodb, @aws-sdk/lib-dynamodb, @aws-sdk/client-secrets-manager, etc.)
  • DynamoDB — identidad de canal, operaciones, llaves referenciadas y telemetría forense
  • Amazon EFS — montaje de CA, CRL, certificados y claves de cliente para Conectividad Segura (TLS / mTLS)
  • AWS Secrets Manager — passphrase de clave privada para cliente TLS / mTLS (nunca en variables de entorno)
