Misión del tld-validador-proxy

Soberanía tecnológica y seguridad proactiva para la validación de canales

Portal técnico: un solo punto de salida controlado hacia validadores — cifrado híbrido correcto, Conectividad Segura (TLS / mTLS) con CA y CRL en EFS, identificación del proxy ante el validador con certificado de cliente mTLS cuando el canal lo exige, identidad de negocio en DynamoDB y trazas sin filtrar secretos.

Lectura recomendada para stakeholders

¿Por qué este proyecto? — Valor y justificación

Además del contexto legacy (tld-api-cuenta-nombre, tld-util, bifurcación axios / CA Telered) y del cifrado de aplicación centralizado, la página concentra tres pilares que explican la Lambda:

  • Corrección del flujo de validación CRL (Seguridad proactiva: revocación antes del envío del body).
  • Identidad del Proxy mediante mTLS (certificado de cliente hacia el validador, perfil por canal).
  • Dashboard de Salud de Canales (Observabilidad forense por fase de Conectividad Segura TLS y mTLS, CRL, HTTP y PROXY_ERR_*).

Profundiza en hallazgos, diagramas comparativos y telemetría en el documento enlazado.

Ir a valor y justificación

Nuestra Misión: Garantizar la integridad y confidencialidad en el transporte de datos financieros, eliminando la deuda técnica de validaciones reactivas y estableciendo una identidad digital soberana mediante mTLS nativo y observabilidad forense.

Garantía de entrega: Centralización de la lógica de reintentos para flujos críticos de inactividad y notificaciones, eliminando la fragilidad de los componentes legacy (inactividad → cuenta-nombre → tld-notificacion) y tratando el Proxy como buffer de resiliencia con backoff exponencial antes de devolver fallo definitivo al originador. Detalle en Valor y justificación — resiliencia.

El proxy es el único punto de salida controlado hacia los canales validadores remotos: centraliza cifrado híbrido (RSA/KMS + AES), Conectividad Segura (TLS / mTLS) con CA en EFS, comprobación de CRL y la resolución de identidad desde DynamoDB, de modo que los microservicios satélite no repitan ni omitan parámetros críticos. No solo valida al validador con la cadena corporativa: cuando el canal lo define, el proxy presenta su propio certificado de cliente para que el socio remoto autentique nuestra salida igual que nosotros autenticamos la suya.

Secuencia maestra: orquestación + telemetría lateral

Cada fase crítica deja huella en Telemetría (DynamoDB): no bloquea el camino feliz si falla el PutItem (política fail-safe), pero en condiciones normales reconstruyes el incidente como si leyeras la cinta negra junto al portero. 

sequenceDiagram
    autonumber
    box LightBlue Orquestacion
    participant Inv as Invocacion
    participant Log as Logger
    participant Idn as IdentityManager
    participant Op as OperacionesPaquete
    participant Htt as httpsClient
    participant Tel as Telemetria_emit
    end
    box Honeydew Validador_remoto
    participant Rem as Endpoint_validador
    end

    Inv->>Log: evento y contexto
    Log->>Tel: emit Logger meta
    Log->>Idn: resolveChannel DynamoDB
    Idn->>Tel: emit Identity fase
    Idn-->>Log: DTO canal inmutable
    Log->>Op: cerrarPaquete
    Op->>Tel: emit Crypto_App
    Op->>Htt: body cifrado y perfil canal TLS mTLS
    Htt->>Tel: emit Handshake CRL
    alt Handshake CRL y revocacion OK
      Note over Htt,Rem: Conectividad segura TLS y mTLS certificado cliente si aplica
      Htt->>Rem: POST body cifrado
      Rem-->>Htt: respuesta HTTP
      Htt->>Tel: emit HTTP_Response
      Htt-->>Op: bytes y status
    else Fallo handshake CRL o revocacion
      rect rgb(255, 160, 160)
        Htt->>Htt: ABORTA sin body hacia validador
        Htt->>Tel: emit error PROXY_ERR
      end
      Htt-->>Op: error controlado
    end
    Op-->>Log: abrir paquete o error
    Log-->>Inv: respuesta API

La Lambda actúa como director de orquesta: IdentityManager sustituye el legado de canal.js con lecturas GetItem/GSI y DTO congelado (nadie reescribe el pasaporte a mitad del control); httpsClient concentra el portero de Conectividad Segura (TLS / mTLS), la CRL y el doble pasaporte mTLS: el validador prueba su identidad hacia nosotros y, cuando el perfil del canal lo exige, nosotros presentamos certificado de cliente hacia él. Entre ambos, OperacionesPaquete llena las cubetas de cifrado de aplicación antes de que un solo byte sensible cruce el cable hacia el validador. El porqué estratégico del proyecto (auditoría, riesgos legacy, centralización de cifrado y telemetría) está concentrado en Valor y justificación.

Alcance

  • Orquestación: dos lecturas de canal (emisor / validador), cifrado hacia el remote endpoint, HTTP(s) con política unificada de Conectividad Segura (TLS / mTLS).
  • Telemetría técnica en DynamoDB (tabla dedicada) para handshake, CRL, mTLS, HTTP y tiempos.
  • Compatibilidad de despliegue con el estándar SAM de los repos existentes (VPC, EFS, KMS MRK).

Stack tecnológico

  • Runtime: Node.js 20.x (LTS)
  • AWS SDK v3 — clientes modulares (@aws-sdk/client-dynamodb, @aws-sdk/lib-dynamodb, @aws-sdk/client-secrets-manager, etc.)
  • DynamoDB — identidad de canal, operaciones, llaves referenciadas y telemetría forense
  • Amazon EFS — montaje de CA, CRL, certificados y claves de cliente para Conectividad Segura (TLS / mTLS)
  • AWS Secrets Manager — passphrase de clave privada mTLS (nunca en variables de entorno)
