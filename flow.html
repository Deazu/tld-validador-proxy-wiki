Flujo General

Ciclo de vida de una petición al tld-validador-proxy: desde el cliente interno hasta el endpoint remoto del validador, pasando por sanitización de logs, resolución de canal en DynamoDB, cifrado híbrido y túnel TLS con política de confianza (CA/CRL).

Lógica del orquestador (app.js)

El handler de la Lambda es el director de orquesta: no implementa TLS ni AES él mismo; encadena módulos con contratos estables. Las analogías del resto de la wiki siguen aplicando: el portero (TLS + CRL) solo deja pasar al interior después del carné; las cubetas (Buffer) aparecen al leer CA/CRL y al ensamblar la respuesta HTTP; el pasaporte doble (mTLS) es opcional por canal.

Contrato de errores y HTTP: Matriz maestra PROXY_ERR_*. Resolución de canal inmutable: clase IdentityManager.

Pseudo-código detallado (orden estricto de ejecución)

FUNCTION handler(event, context):
  correlationId ← event.headers['x-correlation-id'] OR generar UUID
  telemetry.emit({ correlationId, phase: 'Handler_Start', outcome: 'OK' })

  // 1) Sanitización temprana — nunca loguear secretos crudos
  logger.Print('[proxy] evento recibido', sanitize(event))
  SI validación de forma del evento falla:
     telemetry.emit({ correlationId, phase: 'Validation', outcome: 'FAIL', detail: { internalCode: 'PROXY_ERR_VAL_001' } })
     RETORNAR response HTTP 400 con cuerpo genérico

  // 2) Identidad del canal (DynamoDB) — antes de cifrar o abrir TLS
  TRY:
     canalDto ← await IdentityManager.resolveChannel(event.idCanalValidador)
  CATCH CanalNotFoundError:
     telemetry.emit({ correlationId, phase: 'Identity', outcome: 'FAIL', detail: { internalCode: 'PROXY_ERR_ID_001' } })
     RETORNAR 404
  SI NOT IdentityManager.metodoDisponible(canalDto, event.metodo):
     telemetry.emit({ correlationId, phase: 'Authorization', outcome: 'FAIL', detail: { internalCode: 'PROXY_ERR_AUTH_001' } })
     RETORNAR 403

  // 3) Cifrado de aplicación (sobre de negocio) — OperacionesPaquete
  TRY:
     payloadCifrado ← OperacionesPaquete.cerrarPaquete(origen, event.dataPlana, modoDesdeDynamo)
  CATCH error cifrado:
     telemetry.emit({ correlationId, phase: 'Crypto_App', outcome: 'FAIL', detail: { internalCode: 'PROXY_ERR_ENC_001' } })
     RETORNAR 500

  // 4) Transporte — httpsClient.requestSecure con perfil TLS del canalDto
  urlDestino ← resolver URL desde canalDto + operación (urlOperacion o urlValidador)
  TRY:
     telemetry.emit({ correlationId, phase: 'Handshake_Start', outcome: 'OK' })
     httpResult ← await httpsClient.requestSecure({
       url: urlDestino,
       method: 'POST',
       headers: headersDesdeCanal,
       body: payloadCifrado,
       tlsOptions: derivarDesde(canalDto)   // CA, mTLS, SNI, CRL antes del body
     })
     telemetry.emit({ correlationId, phase: 'HTTP_Response', outcome: 'OK', durationMs: ... })
  CATCH TlsHandshakeError:
     RETORNAR mapear PROXY_ERR_TLS_001 → 502
  CATCH CERT_REVOKED / CrlCheckError:
     RETORNAR mapear PROXY_ERR_CRL_* → 502 o 403 según matriz
  CATCH RemoteEndpointTimeout:
     RETORNAR mapear PROXY_ERR_NET_001 → 504

  // 5) Respuesta al cliente interno
  RETORNAR 200 con cuerpo según contrato API Gateway

Contrato de datos entre piezas (qué se pasa de un módulo al siguiente)

Cada etapa consume un producto claro de la anterior; así se evita que el portero (TLS) reciba algo que no sea ya el “sobre” correcto, y las cubetas (Buffer) solo se llenan donde el transporte las necesita (respuesta HTTP, lectura PEM en disco).

Paso Productor → Consumidor Datos que circulan (contrato)
Entrada API Gateway / cliente → handler Evento JSON: idCanalValidador (id o alias), metodo, dataPlana; cabeceras opcionales (x-correlation-id). Sin modo AES ni rutas de llave en claro.
Identidad IdentityManager → orquestador canalDto congelado (inmutable): atributos de tld-validador-canal, llaveCifrado/llaveDescifrado ya resueltos (objeto KMS o fila de llave + ruta EFS según diseño), algoritmoCifrado, operaciones[], urlValidador, perfil TLS (tlsProfile, rutas mTLS, etc.).
Cifrado app Orquestador → OperacionesPaquete.cerrarPaquete Entrada: origen (ruta o descriptor de llave coherente con el DTO), event.dataPlana (objeto), modo = cadena exacta desde Dynamo (algoritmoCifrado normalizado en código). Salida: payloadCifrado (string o Buffer) listo para el cable.
URL efectiva Orquestador (regla de negocio) urlDestino = urlOperacion de la fila que coincide con metodo si existe; si no, urlValidador del canal. Debe cumplir regex del diccionario de datos.
Transporte Orquestador → httpsClient.requestSecure Opciones: url, method, headers (sin secretos en logs), body: payloadCifrado, y tlsContext derivado del canalDto (CA/CRL EFS, SNI, cert/key mTLS + passphrase desde Secrets Manager si aplica). Salida: statusCode, cabeceras, body como Buffer ensamblado (cubetas).

Cualquier await que falle fuera de los tipos anteriores debe degradarse a PROXY_ERR_INT_001 (500) con mensaje opaco al cliente y detalle técnico solo en telemetría.

Explicación exhaustiva por participante

1. Cliente → Proxy Handler

El invocador entrega identificadores de negocio y dataPlana; no envía el modo AES ni rutas de llaves. El handler valida forma del evento, genera o propaga correlationId y prepara la orquestación.

2. Logger (Print / sanitize)

Desde el primer log útil, el proxy debe usar el módulo de logging refactor (logger.js): solo escribe en consola si process.env.PRINT_LOGS === "on" (criterio estricto documentado en Logger). Cada argumento pasa por sanitize: recorre objetos, enmascara claves en DENY_KEYS (p. ej. token, llavedescifrado, llavecifrado), trunca cadenas largas y evita volcar buffers completos. Así un error temprano no filtra Dynamo ni payloads en CloudWatch.

Alinear con operacionesPaquete.js: allí PRINT_LOGS puede evaluarse con .toLowerCase(); en el proxy conviene una política única documentada para operaciones.

3. Canal (DynamoDB + EFS / KMS)

getCanal(idCanalValidador) resuelve la fila en tld-validador-canal (y tablas satélite). Debe producir, entre otros: urlValidador (HTTPS destino), referencias a llaveCifrado (ARN KMS o id hacia tld-validador-canal-llave), algoritmoCifrado para el motor simétrico, y operaciones para validar metodo. Si la llave es local, el registro de llave aporta típicamente archivo relativo a /mnt/tld-llaves/ para componer el argumento origen de cerrarPaquete.

4. OperacionesPaquete (cifrado)

cerrarPaquete(origen, contenido, modo) serializa el contenido a JSON si hace falta, resuelve la variante por modo: aes-256-gcm → variante A (autenticación GCM, paquete concatenado); cualquier otro modo válido en Node (p. ej. aes-256-cbc) → variante B (formato con separadores .). El modo debe coincidir exactamente con el valor almacenado en Dynamo para ese canal validador.

5. HttpClient (túnel de confianza)

El cliente HTTPS nativo (httpsClient.js) construye un Agent con CA leídas de EFS, ejecuta el handshake, obtiene el certificado del peer, ejecuta la validación CRL antes de enviar el cuerpo HTTP, y entonces envía el paquete cifrado a urlValidador. Si el remote endpoint exige mTLS, se extiende el agente con cert/key desde EFS sin cambiar el diagrama de orquestación.

Validación en AWS (consola, canales reales o dummy ya desplegados)

El contrato de esta wiki asume que el equipo prueba contra la cuenta AWS real (dev/qa): canales validadores ya existentes en DynamoDB, URLs que apunten a validadores reales o endpoints dummy corporativos en la misma VPC, y EFS montado como en producción. No se documenta aquí una capa de mocks ni harness local complejo: el “portero” y el “pasaporte” se validan con el mismo TLS y PKI que verá el usuario final.

Checklist en consola (orden sugerido)

  1. DynamoDB: verificar ítems en tld-validador-canal, tld-validador-canal-operacion y tld-validador-canal-llave para el id/alias que usarás; confirmar que el GSI alias-index devuelve el canal sin Scan (ver diccionario de datos).
  2. Lambda → Test: en la consola de Lambda, pestaña Test, crear un evento de ejemplo alineado con Identidad / ejemplo de evento; ejecutar y revisar la respuesta HTTP mapeada (400/403/404/502/504 según matriz PROXY_ERR_*).
  3. CloudWatch Logs: confirmar que logger.Print solo muestra metadatos sanitizados (PRINT_LOGS según política) y que no aparecen PEM, passphrases ni cuerpos completos.
  4. Tabla de telemetría: en tld-validador-proxy-telemetry, filtrar por correlationId (o GSI forense) y comprobar la secuencia de phase hasta HTTP_Response o FAIL con detail.internalCode.
  5. API Gateway: si el borde es HTTP API o REST, usar la acción Test del método o un cliente interno autorizado hacia la URL de etapa; validar que el timeout del API no corta antes que el del validador remoto.
  6. EFS / PKI: desde un bastión o Lambda de diagnóstico en la misma VPC, comprobar que los archivos referenciados por LIB_RUTA_* y por rutas del canal existen y son legibles por el rol de ejecución.

Canales dummy y validadores reales

Un “dummy” en este contexto es otro servicio HTTPS ya registrado en la tabla de canales (misma red, certificado coherente con la CA montada), no un proceso simulado en el portátil del desarrollador. Para fallos de revocación o TLS, la fuente de verdad es el estado del certificado y la CRL publicada en EFS en ese entorno — igual que en producción. Guía de diagnóstico por síntomas: Troubleshooting transporte.

Estrategia «pies de plomo»

Cada capa asume que la anterior puede fallar de forma segura: sin logs crudos, sin TLS relajado, sin cifrado con firma incompleta. La documentación y el código comparten el mismo contrato para que un ingeniero nuevo no tenga que adivinar parámetros ocultos entre repositorios.
