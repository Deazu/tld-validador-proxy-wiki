Sistema de Resiliencia — Capa 2 (asíncrona: SQS y worker)

Cuando el SLA del banco o los límites de API Gateway hacen imposible “esperar en silencio” toda la cascada de reintentos, el Proxy cambia de postura: acepta el trabajo, lo deposita en Amazon SQS y un Lambda Worker independiente completa la entrega con la misma disciplina de transporte que la Capa 1.

En esta página
  • Notificaciones 0010 / 0020 y el buffer SQS
  • HTTP 202 Accepted y desacoplamiento
  • Lambda Worker y httpsClient
  • Sin invocaciones recursivas entre Lambdas

Flujo de notificaciones (0010, 0020) y Amazon SQS

Los métodos de negocio que hoy representan notificación hacia validador — en la familia de Alias (0010) y P2M (0020) — son candidatos naturales a la Capa 2 cuando se elige modo asíncrono en datos. El mensaje que sale del API del Proxy no va “directo al vacío”: se serializa como trabajo idempotente (identificadores de correlación, metadatos de canal, huella del payload ya cifrado o referencia según diseño) y se publica en una cola SQS dedicada al dominio del Proxy.

SQS actúa como buffer de presión: absorbe picos cuando muchas Lambdas satélite disparan notificaciones a la vez; retiene el trabajo si el validador remoto está en mantenimiento; y permite políticas de DLQ para análisis forense sin perder el rastro del intento. Es la buzonería de mensajería certificada entre el momento en que el negocio “dijo enviar” y el momento en que el portero del cable confirma la conversación con el banco.

HTTP 202 Accepted — liberar al cliente al instante

En modo asíncrono, la función del API Gateway responde 202 Accepted en cuanto el trabajo queda comprometido de forma durable: típicamente tras SendMessage exitoso a SQS (o equivalente transaccional que el equipo defina). El cuerpo de la respuesta incluye un correlationId / requestId estable para trazabilidad; no promete que el banco ya haya acusado recibo.

Ese contrato desacopla la latencia percibida del originador (inactividad, alias, p2m) de la latencia real del downstream, que puede incluir varios ciclos de backoff y minutos de ventana de indisponibilidad. El cliente deja de ser rehén del timeout de su propia Lambda o del API interno.

Lambda Worker: mismo httpsClient, backoff exponencial

El Worker es una función Lambda (o conjunto de funciones) distinta del handler síncrono del API: su evento de entrada es el mensaje SQS, no el JSON público del Gateway. Su responsabilidad es reconstruir el contexto mínimo (canal, operación, payload), reutilizar exactamente el mismo módulo httpsClient.js que la Capa 1 y ejecutar la política de reintentos con exponential backoff y jitter entre intentos.

Así se preserva una sola implementación del portero (TLS, CRL, mTLS): no aparece un segundo cliente HTTP “para la cola” con reglas distintas. La telemetría emitida por el worker reutiliza las mismas fases y códigos que ya entiende el dashboard de salud.

Prohibido el “ping-pong” entre Lambdas

El worker no invoca de nuevo al API del Proxy para reintentar, ni encadena Invoke de una Lambda sobre otra en bucle. Los reintentos ocurren dentro del propio proceso del worker (o mediante reentrega controlada por SQS visibility timeout, según el diseño final que el equipo fije), siempre usando httpsClient hacia el validador. Ese límite arquitectónico evita coste explosivo, correlaciones rotas y profundidad de pila impredecible en X-Ray.
