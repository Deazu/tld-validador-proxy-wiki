Arquitectura

1. Identidad y alcance

El servicio concentra las salidas HTTPS hacia los validadores bancarios y absorbe capacidades hoy repartidas en varios repositorios. Referencia explícita de consolidación: tld-api-alias (flujos de inactividad), tld-api-cuenta-nombre, tld-api-p2m, tld-api-r2p, tld-notificacion y utilidades en tld-util. Objetivos transversales: mTLS homogéneo y reintentos gobernados por datos, migrando la lógica de resiliencia asociada a tld-notificacion hacia este proxy cuando corresponda al roadmap.

2. Borde de consumo

Los microservicios consumidores invocan la función del proxy con el SDK de AWS (lambda:InvokeFunction) usando el ARN o el nombre estable de la Lambda. No existe exposición HTTP de entrada gestionada por este producto para el tráfico de negocio hacia el validador.

3. Modelo por defecto: asíncrono (SQS + worker)

La línea base de despliegue prioriza el desacople temporal: el componente de entrada valida, resuelve canal desde DynamoDB, prepara el trabajo y lo publica en Amazon SQS. Una segunda función Lambda (worker de reintento / procesamiento) consume la cola, ejecuta la llamada saliente con el mismo cliente TLS y aplica backoff según política. Así se alinea el SLA percibido del llamador con la duración real del downstream bancario.

El retorno de la invocación al proxy puede materializar una aceptación lógica equivalente a “202” en el cuerpo JSON de respuesta de Lambda cuando el encolado fue duradero, sin implicar acuse del banco.

4. Capa síncrona (opt-in)

Cuando la operación y DynamoDB lo permiten, el mismo binario de orquestación puede ejecutar uno o más intentos HTTP en la misma invocación antes de declarar fallo o delegar a cola. Esta capacidad es complementaria, no sustituto del modelo asíncrono por defecto donde el equipo haya fijado prioridad de producto en SQS.

5. Estructura de repositorio (convención vigente)

tld-validador-proxy/
├── lambdas/
│   ├── tld-validador-proxy/          # Orquestador — punto de entrada Invoke
│   │   ├── app.js
│   │   └── lib/
│   │       └── httpsClient.js        # Único motor TLS + Axios hacia el validador
│   ├── tld-validador-proxy-reintento/ # Worker — evento SQS
│   │   ├── app.js
│   │   └── lib/                      # Misma disciplina de transporte (sin bifurcar stack HTTP)
│   └── layer/
│       └── nodejs/
│           └── package.json          # @aws-sdk/*, axios, dependencias compartidas
├── template.yaml
└── samconfig.toml
Punto único de salida HTTP. Toda negociación TLS/mTLS y peticiones salientes vía Axios deben concentrarse en httpsClient.js (o módulo equivalente único referenciado por ambas funciones), reutilizando el mismo criterio de CRL, agentes y timeouts.

6. Fase previa: documentación

La wiki y los ADR preceden o acompañan el código; la plantilla SAM y samconfig.toml se alinean a repositorios estables de la organización para reducir sorpresas en CI/CD.
