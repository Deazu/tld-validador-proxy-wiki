Cómo activar retryPolicyActive

  1. Localiza la fila de operación en tld-validador-canal-operacion (o el nombre operativo acordado) que empareja idCanalValidador (o GSI por alias) con el metodo — p. ej. 0010 o 0020.
  2. Establece retryPolicyActive en true solo si negocio y el banco aceptan múltiples intentos de entrega hacia ese endpoint (idempotencia o deduplicación en destino).
  3. Completa maxRetryAttempts (véase siguiente sección) y confirma que el documento JSON retryPolicy en tld-validador-canal (o override futuro por operación) define baseDelayMs, maxDelayMs, jitterRatio y retryableHttpStatuses alineados con ese canal.
  4. Para desactivar de inmediato ante un incidente: pasa retryPolicyActive a false y registra el cambio en el ticket de cambio; el Proxy debe comportarse en modo conservador (un intento principal, sin bucle de backoff en Capa 1).
OPT-IN: Si el atributo falta o es false, asume que no hay máquina de reintentos para esa operación, salvo lógica mínima de producto documentada aparte.

Cómo calcular maxRetryAttempts con el SLA del banco

maxRetryAttempts es el número de reintentos después del primer intento (si vale 3, hay hasta cuatro contactos HTTP si todos fallan de forma reintentable). Debe caber dentro de dos límites simultáneos: el timeout de la Lambda del API (Capa 1) o el presupuesto del worker (Capa 2), y el SLA de respuesta que el originador pueda esperar en modo síncrono.

Método práctico:

  1. Fija T_cliente_max = tiempo máximo aceptable que el llamador puede bloquearse (p. ej. 25 s si API Gateway corta a 29 s).
  2. Reserva T_margen para serializar respuesta, logs y telemetría (p. ej. 2–3 s) y T_request estimado por ida y vuelta HTTP al validador (p. ej. 4 s peor caso).
  3. Para cada intento previsto, estima T_backoff(k) como el retardo antes del intento k (backoff exponencial con techo maxDelayMs del JSON de política).
  4. Encuentra el mayor N tal que Σk=0..N T_request + Σk=1..N T_backoff(k) ≤ T_cliente_max - T_margen.
  5. Define maxRetryAttempts = N (reintentos tras el primero) o ajusta a la baja si el banco pide no más de K contactos en ventana de incidente.

En modo 202 + SQS, el cálculo se relaja para el API (solo debe alcanzar a encolar), pero el worker sigue necesitando un tope coherente con DLQ y alarmas. Ver Capa 2.

Ejemplo — método 0010 (notificación Alias)

Objetivo: notificación al validador con reintentos moderados en Capa 1; canal con política JSON ya definida a nivel tld-validador-canal. 

// Ítem en tld-validador-canal-operacion (atributos ilustrativos)
{
  "PK": "CANAL#1000",
  "SK": "OP#0010",
  "metodo": "0010",
  "retryPolicyActive": true,
  "maxRetryAttempts": 3,
  "descripcion": "Notificacion alias / baja — downstream idempotente"
}

En tld-validador-canal para el mismo canal, un retryPolicy coherente podría ser: {"maxRetries":3,"baseDelayMs":400,"maxDelayMs":8000,"jitterRatio":0.2,"retryableHttpStatuses":[408,429,502,503,504]} (el implementador fusiona maxRetryAttempts de la operación con este documento según reglas de producto documentadas en diccionario).

Ejemplo — método 0020 (notificación P2M)

Objetivo: método de notificación validador en P2M; puede activarse Capa 2 si el SLA del API interno es más corto que la ventana de backoff. 

// Ítem en tld-validador-canal-operacion
{
  "PK": "CANAL#2005",
  "SK": "OP#0020",
  "metodo": "0020",
  "retryPolicyActive": true,
  "maxRetryAttempts": 5,
  "asyncDeliveryPreferred": true,
  "descripcion": "P2M — usar 202+SQS en despliegue que exponga worker"
}

El flag asyncDeliveryPreferred es un ejemplo de convención futura: indica a despliegue y orquestador que, si la función y la cola están cableadas, se prefiera HTTP 202 en lugar de agotar backoff en la invocación síncrona. Hasta que exista en el esquema real, el equipo puede documentar el mismo criterio solo con parámetros SAM o feature flag.

Checklist antes de cambiar producción

  • ¿El validador declara idempotencia o clave de deduplicación para 0010/0020?
  • ¿La suma backoff + HTTP cabe en timeout de Lambda y en expectativa del originador (Capa 1)?
  • ¿Alarmas y DLQ de SQS están definidas si se usa Capa 2?
  • ¿Telemetría HTTP_Retry / HTTP_Retry_Exhausted visible en el dashboard de canal?
