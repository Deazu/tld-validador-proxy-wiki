1. Tabla tld-validador-canal

Rol: configuración de negocio y seguridad por canal: quién es el validador en red, cómo se cifra el payload hacia él y qué perfil TLS aplica. Clave de partición típica: idCanal (String). El proxy resuelve el ítem por id o por alias según el evento (ver Identidad).

Atributo Tipo Descripción y ejemplo
idCanal String Partition key. Identificador estable del canal en todo el ecosistema. Ejemplo: CANAL_VALIDADOR_ABC.
alias String Nombre amigable o código corto que el cliente puede enviar en lugar del id; el proxy puede resolver con Scan/GSI según diseño. Ejemplo: validador-nomina.
llaveCifrado String Referencia a la llave de cifrado: ARN de KMS (ej. arn:aws:kms:…:key/…) o identificador de fila en tld-validador-canal-llave según el patrón legado documentado en Identidad.
llaveDescifrado String Referencia simétrica para descifrar respuestas del validador cuando el flujo lo requiera; mismo tipo de convención que llaveCifrado.
algoritmoCifrado String Modo AES acordado para el sobre híbrido. Ejemplos: AES_256_GCM, AES_256_CBC (valores exactos fijados por el equipo y validados en código).
tlsProfile String Perfil de transporte. Decide anclas de confianza: TELERED_PKI (CA/CRL desde EFS según política), PUBLIC (cadena pública estándar), u otros valores acordados. Ejemplo: TELERED_PKI.
mtlsActive Boolean Si es true, el cliente HTTPS del proxy carga cert/key desde las rutas EFS indicadas. Ejemplo: false en la mayoría de canales; true solo donde el validador exija mTLS.
tlsClientCertPath String Ruta absoluta al PEM del certificado de cliente (mTLS), bajo el mount EFS. Ejemplo: /mnt/tld-llaves/certs/proxy-client.pem.
tlsClientKeyPath String Ruta absoluta a la clave privada del cliente (formato PEM u operación acordada). Ejemplo: /mnt/tld-llaves/keys/proxy-client-key.pem.
tlsServerName String Override de SNI cuando el nombre TLS esperado no coincide con el host de la URL. Ejemplo: vacío (usar hostname de urlValidador) o api.interno.empresa.com.
urlValidador String URL HTTPS base o por defecto del validador para este canal. Ejemplo: https://validador.interno/api/v1/validar. Las operaciones pueden sobreescribir con urlOperacion en la tabla de operaciones.

Pueden existir más atributos (auth dinámico, headers, JSON de login, etc.) heredados del legado; este diccionario lista los mínimos para identidad + aplicación + transporte del proxy autónomo.

2. Tabla tld-validador-canal-operacion

Rol: autorización fina: qué códigos de método de negocio (metodo en el evento) están permitidos para un idCanal y, opcionalmente, URL específica por operación. El proxy debe rechazar métodos no declarados antes de abrir TLS o cifrar payload.

Atributo Tipo Descripción y ejemplo
idCanal String FK lógica al canal. Debe coincidir con tld-validador-canal.idCanal. Ejemplo: CANAL_VALIDADOR_ABC.
operacion String Código de operación que el cliente envía como metodo (sort key en diseños compuestos). Ejemplo: 0001 (validación masiva), 0002 (consulta estado).
urlOperacion String Opcional. Si viene informada, sustituye o extiende la URL usada solo para esa operación. Ejemplo: https://validador.interno/v2/validar-masivo. Si vacío, se usa urlValidador del canal.
idCanalValidador / metadatos String / varios Campos auxiliares que el legado use para reporting o dashboard; no son obligatorios para el transporte TLS en sí, pero pueden formar parte del mismo ítem según esquema desplegado.

Clave compuesta típica: PK = idCanal, SK = operacion (o un solo ítem por fila con ambos atributos según el patrón de acceso elegido en SAM).

3. Tabla tld-validador-proxy-telemetry

Rol: observabilidad técnica y forense por petición. Nombre por defecto en código: variable PROXY_TELEMETRY_TABLE o tld-validador-proxy-telemetry (ver telemetry.js). Cada emit escribe una línea de tiempo; no sustituye logs estructurados ni métricas CloudWatch, pero permite reconstruir orden de fases post-incidente.

Atributo Tipo Descripción y ejemplo
correlationId String Identificador único de la petición end-to-end (API Gateway → proxy → validador). Ejemplo: UUID v4 o id generado en el borde. Todas las filas de una misma invocación comparten este valor en consultas forenses.
timestamp Number Epoch en milisegundos (Date.now()) del momento del evento. Ejemplo: 1712345678901. Ordena la secuencia dentro del mismo correlationId.
phase / eventType String En el código actual el campo se llama phase; conceptualmente es el tipo de evento del pipeline. Valores sugeridos: Handshake_Start, CRL_Check, HTTP_Request, HTTP_Response, Error. Ejemplo: CRL_Check.
outcome String Resultado de la fase: por defecto OK en código; en fallos usar FAIL o códigos cortos acordados. Ejemplo: FAIL junto a detail.code = TLS_HANDSHAKE.
durationMs Number Duración opcional de la fase en milisegundos (útil para p95/p99 por etapa). Ejemplo: 87 para un handshake completado.
detail Map / Object Metadatos técnicos sin PII: códigos de error, fase interna, contadores. Ejemplo: { "code": "CERT_REVOKED", "phase": "crl" }. No incluir cuerpos HTTP, DN completos ni tokens.

Recomendación: GSI con correlationId (partition) y timestamp (sort) para consultas por petición. Opcionalmente añadir idCanal dentro de detail para filtrar incidentes por canal sin exponer datos personales.

4. Estructura de llaves en EFS (/mnt/tld-llaves/)

El punto de montaje efectivo en Lambda debe coincidir con los parámetros SAM (LIB_RUTA_ARCHIVO_CA, etc.). La jerarquía lógica siguiente normaliza nombres de carpetas para CA, CRL, certificados de cliente y claves privadas. En despliegues existentes pueden aparecer subcarpetas como tld-ca/ o tld-crl/; lo importante es que las rutas en DynamoDB y en variables de entorno apunten a archivos reales.

Ruta lógica Contenido Uso en el Proxy
/mnt/tld-llaves/ca/ Certificados raíz e intermedios (PEM/DER según convención) Inyectados en https.Agent como ca: [Buffer, Buffer] para confiar solo en la PKI Telered (o subcarpetas equivalentes referenciadas por env).
/mnt/tld-llaves/crl/ Archivos CRL firmados (DER/PEM según parser) Leídos por el módulo de revocación; caché en memoria con TTL para no releer en cada petición.
/mnt/tld-llaves/certs/ Certificados de cliente para mTLS Ruta referenciada por tlsClientCertPath en el canal; solo lectura cuando mtlsActive es verdadero.
/mnt/tld-llaves/keys/ Llaves privadas (.pem u otros formatos acordados) Ruta referenciada por tlsClientKeyPath; permisos POSIX restrictivos; rotación y backup según política de seguridad.

5. Diagrama entidad-relación (lógico)

DynamoDB no impone FK; el diagrama expresa el contrato conceptual entre tablas y el uso de idCanal como eje de unión. La telemetría se vincula al negocio por correlationId y, opcionalmente, por metadatos en detail. 

erDiagram
    VALIDADOR_CANAL ||--o{ CANAL_OPERACION : autoriza
    VALIDADOR_CANAL ||--o{ PROXY_TELEMETRY : traza
    VALIDADOR_CANAL {
      string idCanal
      string alias
      string urlValidador
      string tlsProfile
      boolean mtlsActive
      string llaveCifrado
      string algoritmoCifrado
    }
    CANAL_OPERACION {
      string idCanal
      string operacion
      string urlOperacion
    }
    PROXY_TELEMETRY {
      string correlationId
      number timestamp
      string phase
      string outcome
      number durationMs
    }
Tablas satélite (legado / token): tld-validador-canal-llave (material criptográfico referenciado por id), tld-validador-canal-auth-token (caché de sesión OAuth) y similares siguen fuera de este contrato central pero coexisten en el mismo ecosistema; no son necesarias para describir solo el transporte TLS del proxy.
