1. Tabla tld-validador-canal

Rol: configuración de negocio y seguridad por canal: quién es el validador en red, cómo se cifra el payload hacia él y qué perfil TLS aplica. Clave de partición típica: idCanal (String). El proxy resuelve el ítem por id o por alias según el evento (ver Identidad).

Atributo Tipo Descripción y ejemplo
idCanal String Partition key. Identificador estable del canal en todo el ecosistema. Ejemplo: CANAL_VALIDADOR_ABC.
alias String Nombre amigable o código corto que el cliente puede enviar en lugar del id; en producción debe resolverse con Query al GSI alias-index (ver más abajo), nunca con Scan en caliente. Ejemplo: validador-nomina.
llaveCifrado String Referencia a la llave de cifrado: ARN de KMS (ej. arn:aws:kms:…:key/…) o identificador de fila en tld-validador-canal-llave según el patrón legado documentado en Identidad.
llaveDescifrado String Referencia simétrica para descifrar respuestas del validador cuando el flujo lo requiera; mismo tipo de convención que llaveCifrado.
algoritmoCifrado String Modo AES acordado para el sobre híbrido. Ejemplos: AES_256_GCM, AES_256_CBC (valores exactos fijados por el equipo y validados en código).
tlsProfile String Perfil de transporte. Decide anclas de confianza: TELERED_PKI (CA/CRL desde EFS según política), PUBLIC (cadena pública estándar), u otros valores acordados. Ejemplo: TELERED_PKI.
mtlsActive Boolean Si es true, el cliente HTTPS del proxy carga cert/key desde las rutas EFS indicadas. Ejemplo: false en la mayoría de canales; true solo donde el validador exija mTLS.
tlsClientCertPath String Ruta absoluta al PEM del certificado de cliente (mTLS), bajo el mount EFS. Ejemplo: /mnt/tld-llaves/certs/proxy-client.pem.
tlsClientKeyPath String Ruta absoluta a la clave privada del cliente (formato PEM u operación acordada). Ejemplo: /mnt/tld-llaves/keys/proxy-client-key.pem.
tlsServerName String Override de SNI cuando el nombre TLS esperado no coincide con el host de la URL. Ejemplo: vacío (usar hostname de urlValidador) o api.interno.empresa.com.
urlValidador String URL HTTPS base o por defecto del validador para este canal. Ejemplo: https://validador.interno/api/v1/validar. Las operaciones pueden sobreescribir con urlOperacion en la tabla de operaciones.
retryPolicy String (JSON) o Map Política de reintentos del Proxy hacia el downstream del canal (p. ej. cuenta-nombre / validador). Documentación lógica; el equipo fija el parser en código. Ejemplo de payload JSON serializado: {"maxRetries":4,"baseDelayMs":250,"maxDelayMs":8000,"jitterRatio":0.2,"retryableHttpStatuses":[408,429,502,503,504]}. Campos típicos: maxRetries, baseDelayMs, maxDelayMs, jitterRatio o jitterMaxMs, retryableHttpStatuses, opcional retryOnNetworkError (boolean). Alineado con exponential backoff y OPT-IN por operación: Sistema de Resiliencia — Capa 1 y manual DynamoDB.
fallbackEndpoint String (URL HTTPS) Destino opcional invocado solo tras agotar retryPolicy contra urlValidador / URL de operación, cuando el equipo define un plan B (p. ej. cola o servicio de notificación alternativo) sin reintroducir la lógica ad hoc en Lambdas satélite. Debe ser HTTPS, validado por la misma matriz de URL que el resto del canal; si está vacío u omitido, el Proxy termina la invocación con error mapeado PROXY_ERR_* sin segundo salto automático.

Claves DynamoDB e índices — tld-validador-canal

Concepto Definición recomendada
Partition Key (PK) idCanal (String). Una fila por canal; todas las lecturas “por id” son GetItem o Query de coste O(1) en la partición.
Sort Key (SK) Si la tabla es solo canales, puede omitirse (PK única). Si en el futuro se versionan filas del mismo canal, usar SK fija PROFILE#CURRENT o similar.
GSI1 — alias-index PK = alias (String normalizado a minúsculas en aplicación), SK = idCanal. Proyección: claves + atributos necesarios para el handler. Evita Scan con FilterExpression del legado (canal.js) en caliente.

Pueden existir más atributos (auth dinámico, headers, JSON de login, etc.) heredados del legado; este diccionario lista los mínimos para identidad + aplicación + transporte del proxy autónomo.

2. Tabla tld-validador-canal-llave

Rol: almacena el material o la referencia de archivo para claves simétricas cuando llaveCifrado / llaveDescifrado en el canal no son ARN de KMS. El IdentityManager hace GetItem por id de llave y compone el argumento origen que consume OperacionesPaquete (ruta bajo EFS, etc.).

Atributo Tipo Descripción y ejemplo
idLlave String Partition Key (PK). Coincide con el valor almacenado en tld-validador-canal.llaveCifrado (o descifrado) cuando no es ARN. Ejemplo: LLAVE_CANAL_1000_01.
archivo / material String u otros Ruta relativa al mount EFS o metadatos que el motor de cifrado espera (según implementación de operacionesPaquete en el repo de aplicación).

Claves DynamoDB e índices — tld-validador-canal-llave

Concepto Definición recomendada
Partition Key (PK) idLlave (String). Tabla de acceso por clave simple: una lectura GetItem por referencia.
Sort Key (SK) No requerida si cada llave es un único ítem. Si en el futuro se versionan llaves, usar SK p. ej. VERSION#2026-03-01.
GSI Opcional (p. ej. auditoría por tipo de llave); el camino caliente del proxy solo necesita PK idLlave.

3. Tabla tld-validador-canal-operacion

Rol: autorización fina: qué códigos de método de negocio (metodo en el evento) están permitidos para un idCanal y, opcionalmente, URL específica por operación. El proxy debe rechazar métodos no declarados antes de abrir TLS o cifrar payload. Puede además declarar si la operación participa en la política de reintentos del Proxy (retryPolicyActive, maxRetryAttempts) y, en diseños futuros, modo de respuesta síncrona vs 202 — ver Manual de configuración — Resiliencia y Valor y justificación — §4.5.

Atributo Tipo Descripción y ejemplo
idCanal String FK lógica al canal. Debe coincidir con tld-validador-canal.idCanal. Ejemplo: CANAL_VALIDADOR_ABC.
operacion String Código de operación que el cliente envía como metodo (sort key en diseños compuestos). Ejemplo: 0001 (validación masiva), 0002 (consulta estado).
urlOperacion String Opcional. Si viene informada, sustituye o extiende la URL usada solo para esa operación. Ejemplo: https://validador.interno/v2/validar-masivo. Si vacío, se usa urlValidador del canal.
retryPolicyActive Boolean Si true, el Proxy aplica la política de reintentos hacia el downstream para esta operacion (en conjunto con maxRetryAttempts y el JSON retryPolicy del canal o futuro override por fila). Si false u omitido, comportamiento conservador: un único intento salvo reglas mínimas de red definidas por producto. Útil para distinguir notificaciones 0010/0020 de consultas idempotentes o de solo lectura. Ver Valor y justificación — §4.5.
maxRetryAttempts Number Tope de reintentos después del intento inicial (p. ej. 3 ⇒ hasta 4 contactos HTTP si todos son fallos reintentables). Debe acotarse al tiempo máximo de la Lambda o, en modo asíncrono, al presupuesto del worker/SQS. Si falta y retryPolicyActive es true, el implementador puede heredar maxRetries del documento retryPolicy en tld-validador-canal.
idCanalValidador / metadatos String / varios Campos auxiliares que el legado use para reporting o dashboard; no son obligatorios para el transporte TLS en sí, pero pueden formar parte del mismo ítem según esquema desplegado.

Claves DynamoDB e índices — tld-validador-canal-operacion

Concepto Definición recomendada
Partition Key (PK) idCanal (String). Agrupa todas las operaciones permitidas para un canal.
Sort Key (SK) operacion (String), p. ej. 0001. Acceso: Query con idCanal = :id para cargar el menú de métodos en una sola ronda.
GSI Opcional si existe reporting por operacion global; no es requisito del handler del proxy en el camino feliz.

Clave compuesta típica: PK = idCanal, SK = operacion (o un solo ítem por fila con ambos atributos según el patrón de acceso elegido en SAM).

4. Tabla tld-validador-proxy-telemetry

Rol: observabilidad técnica y forense por petición. Nombre por defecto en código: variable PROXY_TELEMETRY_TABLE o tld-validador-proxy-telemetry (ver telemetry.js). Cada emit escribe una línea de tiempo; no sustituye logs estructurados ni métricas CloudWatch, pero permite reconstruir orden de fases post-incidente.

Atributo Tipo Descripción y ejemplo TTL (Time To Live)
correlationId String Identificador único de la petición end-to-end (API Gateway → proxy → validador). Ejemplo: UUID v4 o id generado en el borde. Todas las filas de una misma invocación comparten este valor en consultas forenses. Purgado con el ítem: 30 días vía atributo expiresAt (estándar de tabla; ver fila expiresAt).
timestamp / timestampIso String (+ opcional Number) Estándar de tiempo: todo instante de evento debe persistirse en ISO 8601 con precisión de milisegundos, p. ej. 2026-03-28T15:04:05.123Z (sufijo Z UTC o offset explícito). Nombre recomendado del atributo: timestampIso o sobrecargar timestamp como String en nuevas versiones del contrato. Si el diseño de Sort Key o GSI exige ordenación numérica, puede añadirse epochMs (Number) derivado del mismo instante, sin contradecir el valor canónico ISO en consultas forenses y export a SIEM. —
phase / eventType String En el código actual el campo se llama phase; conceptualmente es el tipo de evento del pipeline. Valores sugeridos: Handshake_Start, CRL_Check, HTTP_Request, HTTP_Response, Error. Ejemplo: CRL_Check. Purgado con el ítem: 30 días.
outcome String Resultado de la fase: por defecto OK en código; en fallos usar FAIL o códigos cortos acordados. Ejemplo: FAIL junto a detail.code = TLS_HANDSHAKE. Purgado con el ítem: 30 días.
durationMs Number Duración opcional de la fase en milisegundos (útil para p95/p99 por etapa). Ejemplo: 87 para un handshake completado. Purgado con el ítem: 30 días.
detail Map / Object Metadatos técnicos sin PII: códigos de error, fase interna, contadores. Debe incluir internalCode alineado a la matriz (p. ej. PROXY_ERR_TLS_001). Ejemplo: { "internalCode": "PROXY_ERR_CRL_002", "reason": "CERT_REVOKED", "phase": "CRL_Check" }. No incluir cuerpos HTTP, DN completos ni tokens. Purgado con el ítem: 30 días.
expiresAt Number Epoch Unix en segundos (formato que espera DynamoDB para la característica TTL) fijado en cada PutItem a aproximadamente now_s + 30 × 86400. Habilitar en la tabla la opción Time to Live apuntando a este atributo para autolimpieza sin jobs batch: tras ~30 días el servicio elimina el ítem de forma asíncrona, acotando coste y superficie de datos históricos. Sí — 30 días (estándar de retención; ajustable por entorno con gobernanza de datos).

Ciclo de vida: la retención por defecto es 30 días por ítem. Cualquier otro campo de tiempo que se añada al esquema (p. ej. ingestedAt para pipelines downstream) debe seguir igualmente ISO 8601 con milisegundos en su representación intercambio; los consumidores de dashboards leen fechas humanas desde ese formato y, si hace falta, columnas numéricas derivadas para agregaciones.

Claves DynamoDB e índices — tld-validador-proxy-telemetry

Concepto Definición
Estado mínimo (prototipo) El ejemplo en telemetry.js persiste un ítem plano por emit; sin SK explícito dos eventos con el mismo correlationId y timestamp podrían colisionar si la tabla usa PK simple.
Partition Key (PK) — producción correlationId (String) o prefijo estable CID#<correlationId> si se namespacen tablas compartidas.
Sort Key (SK) — producción Obligatorio recomendado: compuesto único por evento, p. ej. ${timestamp}#${phase}#${ulid} o ULID solo, para que múltiples fases de la misma petición no sobrescriban la fila.
GSI — forense por tiempo PK = fecha truncada (YYYY-MM-DD) o ventana horaria, SK = timestamp; proyección de phase, outcome, detail.internalCode. Alternativa: usar Athena sobre export S3 si el volumen crece.
GSI — por canal (opcional) Si detail.idCanal se estandariza, GSI con PK = idCanal, SK = timestamp para incidentes por integración.

Recomendación: GSI con correlationId (partition) y timestamp (sort) para consultas por petición cuando el PK/SK base lo permitan. Opcionalmente añadir idCanal dentro de detail para filtrar sin PII.

5. Restricciones de formato y longitud (validación en borde)

La Lambda debe rechazar datos mal formados antes de DynamoDB o TLS. Los límites siguientes son contrato de ingeniería; ajústalos en esquema JSON del API si usas validación en API Gateway.

Campo / familia Longitud / tamaño Formato (regex u regla)
idCanal, alias 1–128 caracteres ^[A-Za-z0-9_\-]{1,128}$ (sin espacios; normalizar alias a minúsculas antes de GSI).
urlValidador, urlOperacion máx. 2048 caracteres (URL práctica) Esquema https obligatorio en producción: ^https://[A-Za-z0-9\-.]+(:\d{1,5})?(/[A-Za-z0-9\-._~%!$&'()*+,;=:@/]*)?$ (ajustar según política de query strings). Rechazar javascript:, espacios sin codificar y control chars.
tlsServerName (SNI) 1–253 caracteres (límite práctico DNS) Hostname RFC 1123 simplificado: ^([a-zA-Z0-9]([a-zA-Z0-9\-]{0,61}[a-zA-Z0-9])?\.)*[a-zA-Z0-9]([a-zA-Z0-9\-]{0,61}[a-zA-Z0-9])?$ o vacío para “usar host de la URL”.
Rutas EFS (tlsClient*, env LIB_RUTA_*) máx. 1024 caracteres Prefijo obligatorio /mnt/tld-llaves/ (o mount acordado); sin .. ni symlinks salientes: ^/mnt/tld-llaves/[^.\x00]+ (refinar con allowlist de subcarpetas ca|crl|certs|keys|tld-ca|tld-crl).
correlationId 8–128 caracteres UUID v4 o CUID; regex laxo: ^[A-Za-z0-9\-]{8,128}$.
algoritmoCifrado, tlsProfile 1–64 caracteres Allowlist en código: solo valores conocidos (AES_256_GCM, TELERED_PKI, etc.); rechazar el resto.

Los regex anteriores son base; el equipo de seguridad puede endurecer query paths o exigir punycode para IDN.

6. Matriz maestra: códigos internos PROXY_ERR_*

Cada fila de telemetría con outcome: FAIL debe incluir en detail.internalCode (o campo homólogo) uno de estos códigos para correlación con runbooks y alarmas. El cliente HTTP solo ve un status genérico; el detalle técnico vive en DynamoDB y logs redactados. En código, el handler debe propagar objetos de error que expongan un payload estable (p. ej. err.telemetryDetail desde CanalNotFoundError — canal con PROXY_ERR_ID_001, llave Dynamo ausente con PROXY_ERR_ENC_001) para que telemetry.emit no duplique lógica de mapeo.

internalCode HTTP al cliente Fase típica (phase) Descripción para telemetría / runbook
PROXY_ERR_VAL_001 400 Validation JSON del evento inválido: falta idCanalValidador, metodo o dataPlana; tipos incorrectos.
PROXY_ERR_ID_001 404 Identity Canal no encontrado por id ni por alias (GSI); misma semántica que CanalNotFoundError.
PROXY_ERR_AUTH_001 403 Authorization metodo no listado en operaciones o estado != Y para esa operación.
PROXY_ERR_ENC_001 500 Crypto_App Fallo al cerrar paquete (KMS, llave EFS, modo AES incompatible) o ítem ausente en tld-validador-canal-llave para el idLlave referenciado por el canal (GetItem vacío).
PROXY_ERR_TLS_001 502 Handshake_Start / Error Handshake TLS fallido: cadena no confiable, SNI incorrecto, certificado servidor ausente.
PROXY_ERR_CRL_001 502 CRL_Check CRL ilegible, parseo ASN.1 fallido o archivo ausente en EFS.
PROXY_ERR_CRL_002 502 CRL_Check Serial del peer presente en lista de revocación; conexión abortada antes del body HTTP.
PROXY_ERR_NET_001 504 Error RemoteEndpointTimeout: TLS connect o lectura HTTP excedió umbral.
PROXY_ERR_MTLS_001 502 Handshake_Start Certificado o clave de cliente ilegible; passphrase ausente tras Secrets Manager; rechazo del servidor por identidad cliente.
PROXY_ERR_INT_001 500 Error Error no clasificado; revisar stack en logs internos (sin volcar al cliente).

Mapeo exacto HTTP puede variar si API Gateway transforma cuerpos; mantener tabla en SAM/README alineada a esta matriz.

7. Estructura de llaves en EFS (/mnt/tld-llaves/)

El punto de montaje efectivo en Lambda debe coincidir con los parámetros SAM (LIB_RUTA_ARCHIVO_CA, etc.). La jerarquía lógica siguiente normaliza nombres de carpetas para CA, CRL, certificados de cliente y claves privadas. En despliegues existentes pueden aparecer subcarpetas como tld-ca/ o tld-crl/; lo importante es que las rutas en DynamoDB y en variables de entorno apunten a archivos reales.

Ruta lógica Contenido Uso en el Proxy
/mnt/tld-llaves/ca/ Certificados raíz e intermedios (PEM/DER según convención) Inyectados en https.Agent como ca: [Buffer, Buffer] para confiar solo en la PKI Telered (o subcarpetas equivalentes referenciadas por env).
/mnt/tld-llaves/crl/ Archivos CRL firmados (DER/PEM según parser) Leídos por el módulo de revocación; caché en memoria con TTL para no releer en cada petición.
/mnt/tld-llaves/certs/ Certificados de cliente para mTLS Ruta referenciada por tlsClientCertPath en el canal; solo lectura cuando mtlsActive es verdadero.
/mnt/tld-llaves/keys/ Llaves privadas (.pem u otros formatos acordados) Ruta referenciada por tlsClientKeyPath; permisos POSIX restrictivos; rotación y backup según política de seguridad.

8. Diagrama entidad-relación (lógico)

DynamoDB no impone FK; el diagrama expresa el contrato conceptual entre tablas y el uso de idCanal como eje de unión. La telemetría se vincula al negocio por correlationId y, opcionalmente, por metadatos en detail.

    VALIDADOR_CANAL }o--|| CANAL_LLAVE : idLlave_o_ARN
    VALIDADOR_CANAL ||--o{ PROXY_TELEMETRY : traza
    VALIDADOR_CANAL {
      string idCanal
      string alias
      string urlValidador
      string tlsProfile
      boolean mtlsActive
      string llaveCifrado
      string algoritmoCifrado
    }
    CANAL_LLAVE {
      string idLlave
      string archivo
    }
    CANAL_OPERACION {
      string idCanal
      string operacion
      string urlOperacion
    }
    PROXY_TELEMETRY {
      string correlationId
      number timestamp
      string phase
      string outcome
      number durationMs
    }
Otras tablas satélite (legado / token): p. ej. tld-validador-canal-auth-token (caché de sesión OAuth) pueden coexistir en el mismo ecosistema sin formar parte del mínimo del proxy autónomo. La tabla tld-validador-canal-llave está dentro del contrato (sección 2) cuando el canal referencia llaves simétricas por id en lugar de ARN KMS.
